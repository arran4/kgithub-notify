1. **Understand `setupGUI()` behavior**: In KDE's `KXmlGuiWindow`, calling `setupGUI()` clears the existing `QMenuBar` and `QToolBar` and reconstructs them based on the `.rc` file. Since we don't have `.rc` files for the secondary windows (like `NotificationWindow`), their manually created menus and toolbars are being destroyed right after they are created!
2. **Fix `NotificationWindow`**: Move the `setupGUI()` call to the *beginning* of the constructor, *before* the manual menu and toolbar creation. This allows `setupGUI()` to do its base initialization (like state restoration) without destroying the subsequently added widgets. Or, even better, pass a flag to `setupGUI()` like `setupGUI(Default, nullptr)` but standard practice is just to call `setupGUI()` first, then add manual stuff. Alternatively, we shouldn't use `setupGUI()` if we aren't using the XML GUI framework for that specific window, but the user requested "the base functionality of KXMLGUI (like state saving)". So we should call it. Wait, `KMainWindow::setupGUI()` is what normally does the window size saving. In KF5, `KXmlGuiWindow::setupGUI()` reads the xml. `setAutoSaveSettings()` handles state saving. We'll just call `setupGUI()` before the menu creation.
3. **Fix `MainWindow`**: Similarly, in `MainWindow.cpp`, `setupGUI()` is being called *after* `setupMenus()` and `setupToolbar()`, which destroys them! Let's verify by moving `setupGUI()` before `setupMenus()` and `setupToolbar()`. Actually, if we just want `KXmlGuiWindow` as the base class for its `KMainWindow` features without the XML layout, we might not even need `setupGUI()`. But since we already have it, let's move `setupGUI()` before the UI setup in `MainWindow.cpp`. Wait, in `MainWindow.cpp`, `setupGUI()` is currently called at line 74, right *after* `setupMenus()` and `setupToolbar()`. Let's move it to right *before* `setupToolbar()`.
4. **Fix other windows**: Check `TrendingWindow`, `RepoListWindow`, `WorkItemWindow`, `ActionWindow`, `PullRequestWindow`.
   - `TrendingWindow`: `setupGUI()` is called at the end. Menus are created manually in the middle.
   - `RepoListWindow`: `setupGUI()` at the end. Menus created in `setupUI()`.
   - `WorkItemWindow`: `setupGUI()` at the end. Menus created in `setupUi()`.
   - `ActionWindow`: `setupGUI()` at the end. (No manual menus).
   - `PullRequestWindow`: `setupGUI()` at the end. (No manual menus).

Since the user specifically wants the menus to not be missing, moving `setupGUI()` to the top of the window initialization (right after `setWindowTitle` / `resize` / `setCentralWidget`) is the correct approach.
